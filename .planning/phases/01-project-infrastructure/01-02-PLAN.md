Title: `# Design Pillars`

Open with a brief intro: these three pillars are the promises the game makes. Every change must satisfy at least one. If a change violates any pillar, it does not ship.

For each pillar (Understood, Inspired, Less Pressured):
1. **What it means** — 2-3 sentences explaining the pillar in plain terms
2. **What it looks like in the app** — 1-2 concrete examples from the current build
3. **Pillar check (before any change)** — a yes/no question to ask before building. Examples:
   - Understood: "Does this change respond to the user's actual emotional state, or does it impose an assumption?"
   - Inspired: "Does this protect the user's creative momentum, or does it add friction?"
   - Less Pressured: "Does this introduce any time-based mechanic, streak, penalty, countdown, or comparison?"
4. **What violates this pillar** — 2-3 concrete negative examples so Allie can recognize pressure mechanics she might accidentally introduce. Examples:
   - Less Pressured violations: adding a streak counter, showing "days since last check-in", making quest completion time-limited, ranking emotions

End with an "Emotional Acceptance Gate" section: "Before merging any change, ask: does this make the user feel more understood, more inspired, or less pressured? If the answer is no to all three, do not build it."

**docs/dev-reference/conventions.md**:

Title: `# Conventions`

Cover naming and code style rules. Draw from `.planning/codebase/CONVENTIONS.md` but present as a quick-reference guide:

1. **File naming** — components are PascalCase.tsx, utilities are camelCase.ts, hooks are use-kebab-case.ts
2. **Variable naming** — camelCase, booleans start with is/has/should, animated values end with Opacity/Scale/Value
3. **Imports** — use `@/` alias (resolves to project root)
4. **Components** — default exports, props interface at top of file, StyleSheet.create() for styles in same file
5. **Types** — defined in `src/data/types.ts`, use `interface` for objects, `type` for unions

Keep it scannable with examples for each rule.

**docs/dev-reference/contributing.md**:

Title: `# Contributing`

A friendly guide for Allie. Cover:

1. **Before you code** — read the Design Pillars page. Every change must pass the emotional acceptance gate.
2. **Setup** — `npm install`, `npx expo start`, scan QR code with Expo Go
3. **Making changes** — one small change at a time (per implementation rules). Test on your device. If it feels wrong, stop and talk about it.
4. **Voice changes** — any change to Aetherling's dialogue or feedback lines must pass the voice review convention on the Voice page.
5. **Commit messages** — `type(scope): description` format. Types: feat, fix, docs, refactor, test, chore. Keep under 72 chars. Use imperative mood.
6. **The Final Rule** — if a change doesn't make the user feel more understood, more inspired, or less pressured, don't build it.

**docs/dev-reference/git-primer.md** (per D-07):

Title: `# Git Primer`

Written for someone who has used Git minimally. Cover the essentials:

1. **What Git does** — tracks changes to files so you can undo mistakes and work together without overwriting each other
2. **The basic workflow** — `git status` (what changed?), `git add` (stage changes), `git commit` (save a snapshot), `git push` (send to GitHub)
3. **Branches** — what they are (a separate line of work), how to create one (`git checkout -b my-feature`), how to switch (`git checkout master`)
4. **Common situations**:
   - "I want to see what I changed" — `git diff`
   - "I want to undo my changes" — `git checkout -- filename`
   - "I made a mistake in my last commit message" — `git commit --amend`
   - "I need to pull the latest changes" — `git pull`
5. **GitHub** — the website where the code lives. Pull requests are how changes get reviewed before merging.

Keep it short, practical, no theory dumps. Use the actual repo name in examples.
